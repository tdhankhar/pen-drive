fucntionalities we need: 

### Upload 

- basic api in golang for accepting multipart uploads 
- react frontend which can do multipart upload of a file / folder - use some library like uppy 
  - folder uploads should be uploaded with paths as it is on the user-root-folder 
  - user may choose to upload a folder in any subfolder in user-root-folder
  - for each file in the folder, we want to have multipart upload from browser to backend (golang) if file is greather than 5 MB 
  - File duplicatoin flow : 
    - ask user whether to creaate  a new file or replace existing files -> and show a possible list of files that will be impacted .
    - if new file is chosen =>   create a new file with filename_(1).extension style new file.
    - else move original ones to 'trash' in user-root-folder and upload the new ones at the path.
    - above holds true for file and folder uploads both 


### list api 
- s3 list api -- which react app can request and get a list of folder paths on s3 


##### User and other entities 

-  a user can uplaod a file / folder
- there will be a common s3 bucket on backend. and a bucket is 'assigned' to a user (stored in db) 
- user will have a 'root' folder for the user in the s3 bucket 
- any file / folder uploads (referred as 'uploads') will be done in the user-root-folder on s3
- same goes for list api = take note that s3 list api is 'paginated' so we might need to be smart about listing the files to the user. 
- user bucket on s3 = user_id in database, and db has a constraint unique or primary key on the user_id 

### Auth 

- email / password is the auth model we want 
- user can login and frontend does auth via jwt 
- sign up and login pages on frontend 
- the dashboard shown post-login will be a simple file system like interface where user can browser their files (list api) - simple interface for their folder on s3
  - for any file - we send the presigned url for that file (that is used on the 'download' button against filename) 
  - folders can further explroed 
  - any file deleted by the user -> goes to a special 'trash' subfolder in user-root-folder
